How to use the WISPER CLI Library

  • either write an XML configuration (as desccribed here)
  • or configure an instance of com.wisper.cli.CLIParser programmatically using the classes contained within the library

General

This library handles three types of inputs from the command line: Flags, Options and arguments. They are supposed to be arranged in this order:
<executable> [flags and options] arguments...
Name Possible Values Access via CLIParser.ValidatedInput
Flag Set or not set, true or false isFlagSet(String)
Option Any string or any other class, if specified. getOption(String)
Argument Set or not set, true or false getOption(String)

Whether something is a flag, an option or an argument is determined by the prefix. This prefix depends (or at least should depend) on the environment your application is running in. See below for more information on this.

1. Determine/Define CLI Environemnt: Flag and Option markers, escape character

Unix flags are often prefixed with a single hyphen (e.g. -verbose) whereas options are prefixed with two hyphens (e.g. --input-file foo.txt). In DOS/NT however, both are prefixed with a slash (e.g. cd /D D:\).
If someting should contain spaces double-quotes are used to denote the start and end of that (e.g. cd "C:\Program Files\"). If a double-quote should be passed, it has to be escaped. Since the blackslash used in UNIX serves as directory-separater under DOS/NT, ^ is used.

These prefixes and the escape character can be defined individually. The library's class com.wisper.cli.Environment distincts between WINDOWS and UNIX environemnts by default. It will choose the appropriate one for you unless you wish different.

Environment env = new Environment(); // defualt constructor
Environment env = new Environment("--", "/", '\\'); // -- for flags, / for options and \ as escape character

2. Load configuration

Next, an instance of com.wisper.cli.CLIParser has to be created. If you want to configure the parser programmatically the constructors - depending on whether you want to change the environment - will fit:

CLIParser cliParser = new CLIParser(); // new parser with recognized environment, recommended
CLIParser cliParser = new CLIParser(new Environment("--", "/", '\\')); // new parser with special environment

However, if you want to use a XML configuration, you will have to use the appropriate getInstance Method.

CLIParser cliParser = CLIParser.getInstance(
    this.class.getResourceAsStream("cli-config.xml")
);

3. Provide input and access it

Provide input to parse. For more abstract sources you can use the class com.wisper.cli.Input. For most cases this will be the solution:

public static void main(final String[] args)
{
    CLIParser cliParser = /* load the parser however you like */;
    CLIParser.ValidatedInput input;
    try
    {
        input = cliParser.parse(args);
    }
    catch (ParseException ex)
    {
        System.out.println("Failed to initialize: " + ex.getMessage());
        return;
    }
}

Configuring the parser with XML

The root-element of the config xml is an <cli>-Tag. It includes all definitions of flags, options and arguments and their possible values and validation information. A quick example:

<xml version="1.0">
<cli>
    <flag identifier="verbose">
        <alias>v</alias>
    </flag>

    <argument identifier="input" index="0">
        <filter type="file">
            <permissions>READ</permissions>
            <existance>MUST_EXIST</existance>
        </filter>
    </argument>
</cli>

These CLI-Calls are equal:
<executable> -v foo.txt
<executable> -verbose foo.txt

Flags

Flags are configured using the <flag>-tag. It requires the identifier attribute to be set. Note: the content of the identifier-attribute is the string that you will have to use to access the flag with from your sourcecode.
Inside that tag you can specify as many aliases (for abbreviation or compatibility) as you wish. The identifier as well as all the aliases will be recognized.
For example:

<flag identifier="verbose">
    <alias>v</alias>
</flag>

Options

Options are configured using the <option>-tag. It requires the identifier attribute to be set. Note: the content of the identifier-attribute is the string that you will have to use to access the option with from your sourcecode.
Inside that tag you can specify as many aliases (for abbreviation or compatibility) as you wish. The identifier as well as all the aliases will be recognized.
Optionally, you can specify a <default>-tag. This will be the value assumed when the option is not set.
Additionally, you can specify one <filter>-tag that will hint to the parser how to validate input given to that option. See below for filters.
for example:

<option identifier="output">
    <alias>out</alias>
    <alias>o</alias>
</option>

Arguments

Arguments are configured using the <argument>-tag. It requires both the identifier and the index attribute to be set. Optionally, you can set the required-attribute to true. This will cause the parser to throw an error if no input could be mapped to this argument.
The content of the identifier-attribute is the string that you will have to use to access the option with from your sourcecode.
The content of the index-attriute needs to be an integer number greater or equal to 0. It defines what argument should be parsed as defined in the <argument> tag.
Optionally, you can specify a <default>-tag. This will be the value assumed when the argument is not set.
Additionally, you can specify one <filter>-tag that will hint to the parser how to validate input given to that argument. See below for filters.
Example:

<argument identifier="format" index="1" />
<executable> -v foo.txt someFormat

Filters

A <filter>-tag can be specified for options and arguments. It tells the parser how to parse the values for these options/arguments, e.g. integers or files. There are 7 predefined filters (see below). You can also specify another class that is to filter values; it has to implement the com.wisper.cli.filter.ValueFilter interface.

Either the type-attribute or the class attribute can be set on a <filter>-tag. What values are valid for the type-attribute is descibred below:

Number filters: integer, decimal, big-integer and big-decimal

These filters try to parse the value as integer or decimal. You can specify <min>-, <max>- tags and a <radix>-tag for the integer filters; their functionality should be self-explaining.

Filter Value of type-attribute radix supported? Return-Type
Integer-Filter integer yes java.lang.Long
Decimal-Filter decimal no, radix = 10 java.lang.Double
BigInteger-Filter big-integer yes java.math.BigInteger
BigDecimal-Filter big-decimal no, radix = 10 java.math.BigDecimal

Example: 
<filter type="integer">
    <min>0</min>
    <radix>16</radix>
</filter>

File Filter

The file filter returns an instance of java.io.File. Whether or not a file will be accepted can be configured. These criteria are applicable:

  • existance: the file has to exist, the file may not exist or existance is irrelevant
  • permissions: the has to be
    • readable
    • writable
    • executable
  • it has to be a file/directory
  • a specific extension is required

To define the required existance-state use an <exinstance> tag. Its content has to be one of com.wisper.cli.filter.FileFilter.EXISTANCEs values:

  • MUST_EXIST
  • MUST_NOT_EXIST
  • IRRELEVANT

To define the required permissions use a <permissions> tag. Its content has to be one of com.wisper.cli.filter.FileFilter.PERMISSIONs values:

  • READ
  • WRITE
  • EXECUTE
  • READ_WRITE
  • READ_EXECUTE
  • READ_WRITE_EXECUTE
  • IRRELEVANT
Note: on most OS, files that do not exist are also not readable.

To define whether the file has to be a file or a directory use a <type> tag. Its content has to be one of com.wisper.cli.filter.FileFilter.TYPEs values:

  • FILE
  • DIRECTORY
  • IRRELEVANT

To define the required extension use an <extension> tag.

As an example a filter will serve that requires a readable xml file.

<filter type="file">
    <extension>xml</extension>
    <type>FILE</type>
    <existance>MUST_EXIST</existance>
    <permissions>READ</permissions>
</filter>

Set Filter

The set filter allows only values from a given set; case-sensitivity is by default off but can be turned on by a caseSensitive-attribute. Specify the different possibilities with <value> tags. The set filter will always return values from the configuration and never the actual values. For Example:

<filter type="set">
    <value>AES128</value>
    <value>AES192</value>
    <value>AES256</value>
    <value>SERPENT</value>
    <value>THREEFISH</value>
</filter>

This filter is case insensitive. Given the input "ThReEFiSH" it will return the string "THREEFISH".

Regex Filter

The regex filter allows only values matching the given regex. It will return the entire input if matched or a group, if specified by a returnGroup-attribute. Specify the regex using a <regex-tag. For example:

<filter type="regex" returnGroup="1">
    <regex>(\d+)KG</regex>
</filter>

Given the input "15KG" this filter will return the string "15".

Custom Filters

By specifying a class-attribute in the <filter>-tag you can use your own filter-implementation. For example:

package com.myproject;

import com.wisper.cli.filter.ValueFilter;
import com.wisper.cli.ParseException;

public class MyFilter implements ValueFilter
{
    // make sure there is a default-constructor with no arguments!
    
    @Override
    public Object parse(String value)
        throws ParseException
    {
        try
        {
            int val = Integer.parseInt(val);
            if (val % 8 != 0)
            {
                throw new ParseException("needs to be an integer multiple of 8");
            }
            return val;
        }
        catch (NumberFormatException ex)
        {
            throw new ParseException("needs to be an integer.");
        }
    }
}

Rules

You can specify rules that will be checked after all input has been parsed and that can reject that input in case they are not met. For example to require that an option is specified or that two options cannot be set at the same time. Again, you can also use your own implementations.
To refuse all flags and options that are not known by the parser, you will have to add the rule programmatically:

CLIParser cliParser = /* load the parser however you like */;
cliParser.add(Rule.ONLY_KNWON_OPTIONS);
cliParser.add(Rule.ONLY_KNWON_FLAGS);

Almost any other combination can be configured via XML. There are 6 types of rules. Either the type-attribute or the class attribute can be set on a <filter>-tag. What values are valid for the type-attribute is descibred below:

Option rule: option-set

This rule requires that one or more options/flags are set. Specify these options/flags with the <option>-tag. The content of these <option>-tags has to be the identifier-attribute of the defining option/flag. Not that rules should be at the end of the XML-file since referring to a not yet parsed option/flag will result in an error.
Example:

<rule type="option-set">
    <option>input</option>
</rule>

Option rule: option-xor

Basically, this rule could be realised with a xor-rule and nested option-set rules. However, the error message produced by an option-xor rule is much more user-friendly. This rule is intended if options and flags cannot be set at the same time. For example a verbose- and a quiet-flag at the same time do not make much sense:

<flag identifier="verbose" />
<flag identifier="quiet" />
<rule type="option-xor">
    <option>verbose</option>
    <option>quiet</option>
</rule>

Note: you can also refer to arguments here.

Logical rule: and, or, xor and not

These 4 rules connect other rules with the respective logical operator. For example: If a logfile should be written (log flag), then a logfile has to be specified (logfile option). In a later release there will be an <error>-tag to allow for more user-friendly error-messages on rule-fail.

<flag identifier="log">
    <alias>l</alias>
</flag>
<option identifier="logfile">
    <alias>lf</alias>
    <filter type="file">
        <permissions>WRITE</permissions>
    </filter>
</option>

<rule type="xor">
    <rule type="not">
        <rule type="option-set">
            <option>log</option>
        </rule>
        <rule type="option-set">
            <option>logfile</option>
        </rule>
    </rule>
    <rule type="option-set">
        <option>log</option>
        <option>logfile</option>
    </rule>
</rule>

Own implementations

You can implement your own rules by specifying a class-attribute instead of a type-attribute and implement the correct interface. There are two ways of doing so:

Simple Rule

Implement com.wisper.rule.Rule. If the specified class implements this interface the <rule>-tag my not have any contents and the class has to have a constructor without arguments.

<rule class="com.myproject.SimpleRule" />

Combined Rule: combine multilple rules

Implement com.wisper.rule.CombinedRule. If the specified class implements this interface the <rule>-tag may have nested <rule>-tags and the class has to have a (Rule[]) constructor. Needless to say, the defined rules from the XML document will be passed to your class when constructing an instance.

<rule class="com.myproject.CombinedRule">
    <rule type="option-set">
        <option>input</option>
    </rule>
    <rule type="option-set">
        <option>output</option>
    </rule>
</rule>
